# EVENTS

## Purpose

Capture the future direction for Game Night events once a single event can contain multiple playable experiences inside it.

This is a product-spec note, not an implementation plan. It exists so current event work stays compatible with the larger structure that will come later.

## Core Idea

One top-level event can contain multiple game-specific units inside it.

Examples:

- A board game night with separate tables for different games
- Multiple tables for the same game at different skill levels
- A larger social event with several parallel sign-up experiences

The top-level event is the umbrella social object. The child units are the actual seatable experiences guests join.

## Working Terms

Current naming direction:

- Parent: `Event`
- Child: `Table`, `Session`, or `Subevent`

`Table` is the most natural term for board games, but the data model should stay flexible enough that product copy can change later.

## Product Intent

The app should eventually support:

- one public or private event page
- multiple game tables inside that event
- separate seat limits per table
- separate participant lists per table
- multiple tables for the same game
- skill-tiered tables such as beginner and intermediate

This is the right shape for the future search case where a player looks for open events or open seats for a specific game.

## Why The Current Model Will Break

Today one event mostly behaves like one attendee pool with one shared game context. That is too flat for the future state.

It breaks down when:

- one event has multiple simultaneous games
- seat caps need to apply to a specific table rather than the whole event
- players need to choose a game table after RSVP
- the same game appears more than once in the same event
- search needs to understand open seats at the table level

Because of that, the current `EventGame` relationship should not be stretched into a seating system.

## Future Product Shape

### Parent Event

Owns:

- title
- description
- cover image
- venue and address rules
- event-wide time or time window
- visibility (`public` vs `private`)
- RSVP deadline and overall RSVP state
- invite list and event-level attendance counts

### Child Table / Session

Owns:

- game
- table title or label
- skill level
- optional host or table lead
- min and max seats
- participants
- waitlist
- optional start/end override
- optional notes

## Relationship Rules

- One event can contain many child tables.
- One child table belongs to exactly one event.
- Guests RSVP to the event first.
- Table selection is a second step after or alongside RSVP.
- Child tables inherit parent visibility by default.
- A guest may eventually be allowed to join zero, one, or multiple tables depending on later product rules.

## Public / Private Model

This future structure should keep the event-level visibility model already discussed:

- `private`
  - exact address stays hidden until RSVP
  - custom location label can still be shown
  - event remains invite-oriented

- `public`
  - anyone can view core event details
  - full address is visible
  - guest list identities stay hidden unless the viewer has RSVP'd
  - counts such as going or interested can remain visible

Important: visibility belongs to the parent event first. Child tables should not invent a second, conflicting privacy system unless there is a very strong future reason.

## Discovery Implications

Discovery is not the next feature, but this structure should prepare for it cleanly.

Later, public events should be discoverable and browsable in ways such as:

- browse public game nights nearby
- find public events that include a specific game
- find open beginner tables for a game
- find events with seats still available

That means future indexing and filtering will likely need both:

- parent event metadata
- child table metadata including game, skill level, and seat availability

## RSVP vs Table Signup

These should remain separate concepts.

- RSVP answers whether someone is attending the overall event.
- Table signup answers where they are playing within that event.

This distinction matters because someone may:

- RSVP before choosing a table
- change tables later
- attend socially without claiming a seat immediately

## Suggested Future Data Model

Parent layer:

- `events`

Child layer:

- `event_tables` or `event_sessions`
- `event_table_participants`
- optional `event_table_waitlist`

Likely child fields:

- `event_id`
- `game_id`
- `title`
- `skill_level`
- `host_user_id`
- `min_players`
- `max_players`
- `starts_at`
- `ends_at`
- `sort_order`

## UX Direction

Future host flow:

1. Create the umbrella event.
2. Set visibility, location, and RSVP rules.
3. Add one or more game tables.
4. Set seat caps and labels for each table.
5. Publish the event.

Future guest flow:

1. Open or receive the event.
2. RSVP to the overall event.
3. Review available tables.
4. Claim a seat or join a waitlist.

## Design Guidance For Current Work

Current event work should stay compatible with this model by following these rules:

- keep visibility and address privacy at the event level
- do not assume one event equals one playable seat pool
- do not overload `primary game` or `EventGame` with seating semantics
- keep permission helpers generic enough to work for both event pages and future table sections
- keep RSVP deadline modeled at the event level unless table-level cutoffs become clearly necessary later

## Recommended Rollout Order

1. Ship strong parent-event behavior first:
   public/private visibility, RSVP deadline, location privacy, guest-list privacy.
2. Add richer event editing without discovery:
   make the parent event more complete before exposing browse surfaces.
3. Introduce child tables:
   game-specific seats, participant lists, and waitlists.
4. Add discovery later:
   browse public events, search by game, search by open seats.

## Open Questions

- Can a guest sit at more than one table inside the same event?
- Are table signups single-choice or multi-choice?
- Can tables run sequentially as well as in parallel?
- Should table hosts differ from the parent event host?
- Do tables need their own chat, notifications, or updates?
- Do any public events need invite-only child tables, or is that too complex?

## Recommendation

Keep building the current product around a strong parent event model. When multi-table events arrive, they should extend that model, not replace it.
