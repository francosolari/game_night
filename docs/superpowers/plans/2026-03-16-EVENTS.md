# EVENTS

## Purpose

Capture the future product direction where one larger event can contain multiple playable subevents, tables, or tracks with their own participants and seat limits.

This is not the design to implement now. It is a planning document so current event-visibility work does not block or distort this future structure.

## Working Concept

One parent event can contain multiple child playable units.

Examples:

- A board game night with separate tables for different games
- The same game hosted at multiple skill levels such as beginner and intermediate
- A drop-in social event where each table has a host, seat cap, and waitlist

The parent event is the umbrella social object. The child units are the actual seatable game experiences.

## Naming Direction

Current working terms:

- Parent: `Event`
- Child: `Table`, `Session`, or `Subevent`

Open question:

- `Table` is intuitive for board games
- `Session` is more general
- `Subevent` is structurally accurate but less friendly

For now, design future systems so the internal model can evolve without locking product copy too early.

## Why This Matters

Today the app treats one event as one player pool plus one main game decision. That works for simple invites, but it breaks down when:

- an event has multiple games at the same time
- each game has its own seat count
- guests need to sign up for a specific game experience
- hosts want beginner and advanced tables separately
- search should eventually surface open seats for a specific game

This means the future system should not overload the current `games` list or `primary game` concept to represent actual seatable participation.

## Proposed Future Shape

### Parent Event

Responsible for:

- title
- description
- cover image
- overall venue
- event-wide time or time window
- visibility and discovery
- overall guest access rules
- global RSVP

### Child Table / Session

Responsible for:

- linked game
- host or table lead, if different from parent host
- skill level or label
- seat minimum and maximum
- participant list
- waitlist
- optional start/end time override
- optional notes

### Relationship Rules

- A parent event can have many child tables.
- A child table belongs to exactly one parent event.
- Child tables inherit event visibility by default.
- A guest can RSVP to the parent event without immediately selecting a child table.
- A guest can later claim a seat in one or more allowed child tables depending on future rules.

## Search / Discovery Implications

This structure is the correct base for future search such as:

- find public events containing `Terraforming Mars`
- find open beginner `Root` tables nearby
- browse events with seats still available

Search should eventually index both:

- parent event attributes
- child table attributes such as game, seat availability, and level

## Why Not Build This Into Current Event Games

The current `EventGame` attachment is lightweight metadata. It does not represent a seatable unit with participants and capacity.

Trying to stretch it into that role would create unclear ownership around:

- who is seated where
- whether seat caps apply to the event or the game
- how waitlists behave
- how multiple tables of the same game differ

That should become a dedicated model later.

## Suggested Future Data Model

### Parent

- `events`

### Children

- `event_tables` or `event_sessions`
- `event_table_participants`
- optional `event_table_waitlist`

Possible fields for child rows:

- `event_id`
- `game_id`
- `title`
- `skill_level`
- `min_players`
- `max_players`
- `sort_order`
- `starts_at`
- `ends_at`
- `host_user_id`

## Product Flow Direction

Potential future host flow:

1. Create the umbrella event
2. Add one or more tables/sessions
3. Assign a game and seat cap to each
4. Publish event
5. Guests RSVP to the event
6. Guests join a table or are assigned to one

Potential guest flow:

1. Discover or open event
2. RSVP to the parent event
3. See available tables
4. Join an open table or waitlist

## Compatibility Guidance For Current Work

Current event-visibility implementation should:

- live at the parent event level
- avoid location or privacy logic that assumes one game equals one participation slot
- keep permission helpers generic enough to work for both parent events and future child tables

Current RSVP deadline work should also be modeled at the event level first, with the option to add per-table cutoffs later only if needed.

## Open Questions For Later

- Can one guest hold seats at multiple tables in one event?
- Does RSVP happen separately from table signup, or are they combined?
- Are tables sequential, parallel, or both?
- Can a public event contain private or invite-only child tables?
- Do tables have their own chat, updates, and notifications?
- How should beginner/intermediate labels be normalized?

## Recommendation

Do not merge this future structure into the current event implementation yet.

First ship:

- event-level visibility
- private/public privacy rules
- RSVP deadline

Then design the parent/child event system as a dedicated follow-up project once discovery and seat-based search become active priorities.
